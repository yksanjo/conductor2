# Conductor V3 — delegate

> V1 **watches** windows you opened. V2 **launches** fleets you design. Both still assume
> you're sitting there. V3 is the version that works while you don't: **standing missions,
> verified outcomes, and one decision inbox.**

## The arc

| | verb | you are | the unit |
|---|---|---|---|
| V1 | watch | at the keyboard, juggling windows | a session |
| V2 | launch | at the keyboard, firing swarms | a swarm |
| V3 | delegate | **gone** — asleep, at a meeting, on a plane | a **mission** |

V1 and V2 solved the mechanics: reading trails nobody else reads, driving TUIs through the
only channel that exists (`tmux send-keys`), verified delivery, pane-stage state machines,
an eval harness that measures whether coordination actually happens. V3 composes those
mechanics into the thing a solo founder actually needs: **a fleet that runs on a schedule,
proves its own work, and interrupts you only for decisions that are genuinely yours.**

## The three pieces

### 1. Missions, not sessions

A mission is a file — same spirit as a V2 preset, plus a lifecycle:

```
mission: nightly-repo-health
trigger: cron 03:00            # or: on-event (CI red, new issue, price move)
crew: web3-security preset, 4 agents
budget: 60 min / N tokens
verify: npm test && gitleaks detect   # "done" = this passes, not "agent says done"
on-fail: retry once, then escalate
escalate-to: decision inbox
```

The conductor fires the swarm (V2 `fire()`), babysits it (V2 stall/menu/kickoff detection),
and **runs the verifier itself**. An agent claiming success is evidence; a green verifier is
proof. This is V2's eval-harness philosophy promoted from "do agents coordinate?" to "did
the mission produce the artifact?"

### 2. The decision inbox

V1's irreversibility gate (deploy · send · delete · spend) already decides *what* must
escalate. V3 gives every escalation one place to land: a single queue across **all**
missions and fleets, each item carrying its evidence — the diff, the test output, the
draft post, the simulated transaction.

You don't wake up to 50 windows. You wake up to:

```
☐ ship conductor 0.7.2        evidence: 222 tests green, diff attached
☐ post the launch thread      evidence: draft + meme attached
☐ rotate the leaked key       evidence: probe shows it still live
```

Approve / deny / edit from anywhere the queue can reach you — the cockpit, or a phone via
a notifier (Telegram is the obvious first transport). The gate stays a **gate**: V3 never
auto-approves an irreversible action, it just makes saying yes take five seconds instead
of a context switch.

### 3. The calibration ledger

V2 measures one config's reliability. V3 keeps score across missions: which preset ×
topology × crew size actually delivers which kind of work, at what cost. Brier-style
honesty — "no edge" is a valid result — so the conductor can say "for review missions,
the 3-agent pipeline beats the 5-agent hierarchy and costs 40% less" with receipts, and
eventually pick the crew for you.

## What V3 is NOT (honest seams, stated up front)

- **Not autonomy over irreversible actions.** The inbox makes human approval cheap; it
  never removes it. That asymmetry (false gate = one tap; false pass = a bad deploy or
  real money) is the design center, inherited from V1's policy gate.
- **Not a cloud service.** Same zero-dep, runs-on-your-machine posture. A standing daemon
  (or a $35 Pi) is the deployment story, not a hosted control plane.
- **Not a guarantee.** Coordination stays probabilistic (prompt-level, not mechanical) —
  V3 manages that honestly with verification + retries + escalation, exactly as V2's
  README already frames it.

## Smallest real V3 (the wedge)

One standing mission, end to end:

1. `conductor3 mission add` — mission file + cron entry.
2. Fire via V2's engine; watch via the board; verify via the mission's check command.
3. Escalations → one JSONL queue + a Telegram notifier with approve/deny replies.
4. A ledger line per mission run: crew, duration, verifier result, tokens.

Everything above exists as a part today — `fire()`, the gate, verified delivery, the eval
watchdog, systemd on the Pi. V3 is the composition, not new physics.
